项目概览

本项目对 Yelp 学术数据集（5 万条评论）构建了一个四阶段递进式 NLP 实验流水线。 每一阶段在前一阶段的基础上延伸，系统观察当方法从传统特征工程逐步演进至具备自主推理能力的 检索增强 Agent 时，模型性能与能力的变化规律。

核心演进路径：特征工程 → 固定预训练语义表示 → 端到端 Transformer 微调 → 大模型直接提示推理 → Agentic 检索增强系统

第一阶段 — Baseline 对比

在 5 分类星级预测任务上，对三种建模范式进行对比（测试集结果）。

模型AccuracyMacro F1
TF-IDF + Logistic Regression56.92%56.56%
Sentence-BERT + Logistic Regression53.25%52.89%
BERT（微调）62.50%62.50%

为什么微调 BERT 效果最好？端到端训练让文本表示层和分类头针对具体任务联合优化，每一层的参数都可以根据星级预测目标进行调整。固定表示在推理时是不可变的，预训练目标与星级分类之间的不匹配无法被弥补。

为什么 Sentence-BERT 反而不如 TF-IDF？Sentence-BERT 的嵌入是为句子对语义相似度任务设计的，而非单句分类。星级评分与特定情感词汇（"terrible"、"amazing"、"overpriced"）高度相关，TF-IDF 的词频统计能直接捕捉这类信号；而 Sentence-BERT 将其压缩为一个为其他目的设计的稠密向量，反而损失了关键的判别信息。

Off-by-1 准确率（容差精度）说明：在星级评分预测任务中，若模型预测的星级与真实标签相差不超过 1 星（例如真实为 5★，预测为 4★ 或 3★ 均不算作 Off-by-1 错误），则视为"容差正确"。 这一指标捕捉了实际可接受的近似预测——将 4★ 商家预测为 3★ 与预测为 1★ 在实际意义上有本质区别。 所有模型的 Off-by-1 准确率均高达 97% 以上，这反映了相邻星级之间的语言信号本身存在模糊性， 是评分预测任务的固有难点，而非模型能力的不足。

为什么 RoBERTa 优于 BERT 和 DistilBERT？RoBERTa 在预训练时去掉了 NSP（下句预测）任务，采用动态 Masking 策略，并使用了更大规模的数据进行训练，生成的词表示更加鲁棒，迁移到下游任务时具有更好的起点。DistilBERT 虽然推理更快，但蒸馏过程中不可避免地损失了部分表达能力。

为什么 LR=1e-5 和 batch=4 效果最好？较小的学习率让预训练权重得到更温和的更新，减少灾难性遗忘（catastrophic forgetting），避免破坏模型在预训练阶段习得的语言知识。小 batch size 引入更多噪声梯度，相当于一种隐式正则化，在数据量适中的微调场景下有助于提升泛化能力、降低过拟合风险。

为什么 5 分类比二分类难得多？二分类只需区分明显的负面（1–2★）和正面（4–5★）评论，情感信号清晰。5 分类则需要区分相邻星级（如 3★ vs 4★），这两类评论在词汇和语气上高度相似。Off-by-1 准确率高达 97% 以上说明模型"方向正确"——真正的瓶颈是相邻等级之间的语义模糊性，这是该任务的固有难点。

最优配置（5 分类）：RoBERTa-base · LR=1e-5 · batch=4 · epochs=4。 该 checkpoint 在第四阶段中被复用为评论分类器工具。

Assistant 回复的训练格式：

<think>
  {模型生成的推理链}
</think>

训练数据只包含基础模型已预测正确的样本——推理链错误的样本会被丢弃，避免强化错误的推理路径。

DeepSeek 提示词优化 — 第二层改进

DeepSeek-R1 需要一个 Qwen 完全跳过的中间提示词优化步骤（第二层）。零样本性能受限于 token 预算不足和缺乏锚定示例。以下改动将 5 分类准确率从 55.60% 提升至 61.56%：

参数/设置基准提示词改进提示词
max_tokens128512
Few-shot 示例无（零样本）5 条示例（每个星级各一条）
推理长度限制无限制"简要推理（2–3 句话）"
有效预测率65.4%~99%
5 分类准确率55.60%61.56%

基准提示词（max_tokens=128，零样本）：

You are a Yelp review rating classifier.
Analyze the review carefully.
The label must be one integer from 1 to 5.
You may reason first, but your LAST line must be exactly:
Final rating: X
where X is one of 1, 2, 3, 4, or 5.

Review: {text}

改进提示词（max_tokens=512，5-shot 示例，简要推理约束）：

You are a Yelp review rating classifier.
Reason briefly (2-3 sentences), then output your final answer.
Your LAST line must be exactly: Final rating: X  (X ∈ {1,2,3,4,5})

Here are 5 examples:
Review: "Absolutely terrible experience..." → Final rating: 1
Review: "Below average, won't return..."   → Final rating: 2
Review: "Decent, nothing special..."        → Final rating: 3
Review: "Really enjoyed it, will be back!" → Final rating: 4
Review: "Outstanding! Best in the city!"   → Final rating: 5

Review: {text}

为什么这些改动能提升准确率？

  1. max_tokens 128 → 512：DeepSeek-R1 的思维链通常需要 100–300 个 token。仅有 128 个 token 时，约 35% 的回复在输出 "Final rating:" 之前就被截断，导致解析失败。将预算提升至 512 直接恢复了这批样本的有效预测。
  2. 5-shot 示例（每个星级各一条）：零样本提示下模型对各星级对应的评论语气缺乏锚定，容易出现星级偏移。每个星级提供一条校准示例，帮助模型统一格式并正确区分星级边界，同时减少格式错误。
  3. "简要推理（2–3 句话）"约束：若无长度限制，思维链仍可能占满 512 个 token 而未输出最终标签。限定推理在 2–3 句以内，确保最终答案始终在 token 预算内出现。

为什么 Qwen 零样本就已接近微调水平？Qwen2.5-7B-Instruct 在预训练阶段见过大量评论类文本和指令跟随数据，本身已具备理解评论情感与星级关系的能力。LoRA 微调的边际增益极小（二分类仅 +0.39%），反映的是该任务对该模型的能力天花板，而非微调方法的局限。

为什么 DeepSeek 零样本效果很差？DeepSeek-R1 是推理模型，需要生成完整的思维链（chain-of-thought）后才能输出最终答案。max_tokens 不足时，推理链在中途被截断，模型从未得出最终标签。这是提示词工程问题，而非模型能力问题——增大 token 预算并加入 few-shot 示例后，5 分类准确率从 55.6% 跳升至 61.6%。详见上方"DeepSeek 提示词优化"模块。

为什么 LoRA 微调能有效提升？LoRA 冻结基础模型权重，仅学习少量低秩更新矩阵，在保留预训练语言知识的同时适应输出格式。对于能力已经很强的大模型，LoRA 的核心作用是格式对齐（如输出单个数字而非一段话），而非能力的根本提升，因此准确率增益稳定但幅度有限。

为什么 DeepSeek 微调要用思维链训练数据？若直接用"输入→标签"对进行微调，会压制模型的链式推理结构，破坏其推理能力。方案是先让模型对训练集运行推理，从预测正确的样本中提取 <think>...</think> 推理链，再以"输入 → 思维链 + 标签"的格式进行训练，从而在保留推理能力的同时提升分类准确率。

两个检索工具有意拆分：让 7B 模型每次只面对一个职责单一的工具，相比合并为一个多功能检索工具， 能显著提升工具选择的可靠性。

技术栈

LangGraph create_react_agent Qwen2.5-7B via Ollama FAISS IndexFlatIP all-MiniLM-L6-v2 RoBERTa-base 分类器 LangChain + langchain-ollama Gradio 5

向量知识库

  • 5 万条 Yelp 评论 → 60,823 个文本块（按句子切分，每块 ≤180 词）
  • 使用 all-MiniLM-L6-v2 编码（384 维，L2 归一化 → 内积等价于余弦相似度）
  • FAISS IndexFlatIP 精确最近邻检索
  • 商家内预过滤检索：仅加载目标商家的嵌入子集做矩阵乘法，避免小商家结果被稀释

评测设置

20 道问题 × 4 种题型 × 3 个系统 = 60 组回答。人工在 5 个维度上评分（每维 0–2 分，满分 10 分）。

题型示例问题

RAG Baseline（6.90/10）：固定的工具调用顺序保证了稳定性，证据引用充分，幻觉率仅 5%，充分证明了检索增强对于直接生成方式的优越性。

Full Agent（4.75/10）：在方面分析和跨商家规律等复杂题型上正确性最高，自主工具选择具有优势。但 7B 模型在 ReAct 提示下的不稳定性导致幻觉率达 25%，效率得分极低（0.10）。

项目亮点

  • RAG 流水线设计：chunk 级 FAISS 索引、商家内预过滤检索、基于证据的结构化综合
  • LangGraph Agent 实现：ReAct 循环、工具绑定（create_react_agent）、消息轨迹提取
  • 面向小模型的工具设计：将检索工具拆分为两个职责明确的工具，显著提升 7B 模型工具选择可靠性
  • 结构化评测体系：5 维度评分标准设计、三方系统对比、幻觉率量化
  • 生产级工程实践：崩溃续跑评测脚本、多层 JSON 容错解析、配置集中管理
  • 可交互 Demo：Gradio 5 应用，支持 160 个商家的可搜索下拉菜单、实时工具调用日志和证据并排展示
  • 系统化 ML 实验：216 组超参数搜索，自动化日志记录与结果聚合（第二阶段）
